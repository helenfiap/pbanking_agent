## The Problem Being Solved

Franq is a marketplace for 150+ financial products. Their **Personal Bankers** match clients to credit and investment products. The copilot lets them query client behavioral data in natural language instead of writing SQL — e.g.:

> "Liste os 5 estados com maior número de clientes que compraram via app em maio."

The agent translates that into SQL, runs it, picks the right visualization, and returns a plain-language answer — all in Portuguese.

## Stack

### Core Framework
| Layer | Choice | Why |
|---|---|---|
| Agent orchestration | **LangGraph** | StateGraph with typed state, conditional edges, retry loops — production-grade agent control flow |
| LLM (primary) | **Gemini 2.5 Flash** via Google AI Studio | Free tier for demo; strong SQL generation |
| LLM (fallback) | **Azure OpenAI GPT-4o** | Optional switch via `LLM_PROVIDER=azure` env var |
| UI | **Streamlit** | Fast iteration; free Community Cloud deployment |
| Database | **SQLite** | Local, zero infra, matches challenge data |
| Logging | **JSONL** (per banker) | Schema-free, pandas-native, Parquet-ready |
| Observability | **LangFuse** | Per-node token tracking, latency, cost — keys set, callbacks pending |
| Package manager | **uv** | 100 packages in ~1.5s vs pip's minutes |

### LLM Strategy: Mixed Model Per Node
Not every node needs GPT-4o. Heavy LLM only where it matters:

| Node | Model tier | Reason |
|---|---|---|
| Schema inspector | No LLM | Deterministic PRAGMA queries |
| Planner | Heavy | Needs full reasoning over schema + question |
| SQL generator | Heavy | SQL quality is critical path |
| SQL executor | No LLM | Pure Python execution + error catching |
| Error recovery | Cheap | Focused task: fix a specific syntax error |
| Visualization agent | Cheap + deterministic pre-filters | Most cases resolved without LLM at all |
| Response formatter | Cheap | Summarization, low complexity |

### Data Pipeline Design
Logs are written as JSONL, designed with a clear upgrade path:

```
JSONL (now) → Parquet (at scale) → DuckDB (analytics) → GCS + BigQuery (production)
```

The agent code never changes when the storage layer upgrades.

## Architecture: 7-Node LangGraph

```
START
  │
  ▼
[1] schema_inspector      # PRAGMA table discovery, no LLM
  │
  ▼
[2] planner               # Heavy LLM — creates a query plan
  │
  ▼
[3] sql_generator         # Heavy LLM — writes SQLite-specific SQL
  │
  ▼
[4] sql_executor          # Runs query, catches exceptions
  │
  ├─ error + retries < MAX ──▶ [5] error_recovery ──┐
  │                            Cheap LLM, fixes SQL  │
  │                                                  │ (loops back to executor)
  ├─ error + max retries ──▶ [7] response_formatter  │
  │                                                  │
  └─ success ──────────────▶ [6] visualization_agent─┘
                               Hybrid: deterministic + cheap LLM
                                │
                                ▼
                             [7] response_formatter
                               Cheap LLM, PT-BR answer
                                │
                                ▼
                              END
```

### Shared State (`AgentState`)
```python
class AgentState(TypedDict):
    question: str
    schema: str
    plan: str
    sql: str
    result: Optional[str]      # JSON string of query results
    error: Optional[str]
    retries: int
    viz_spec: dict             # {type, x, y, color, title}
    response: str
    trace: list[str]           # reasoning steps with latency
```

## Features Built

### 1. Dynamic Schema Discovery
No hardcoded table or column names. On every run, Node 1 calls `PRAGMA table_info` on all tables and builds a full schema string including row counts. The LLM always sees the current state of the database.

### 2. Error Recovery Loop
When SQL execution fails, the agent retries up to `MAX_RETRIES` times. The error recovery node receives the exact error message from SQLite and rewrites only the failing query — a cheap, focused LLM task. After max retries, it fails gracefully with a PT-BR explanation.

### 3. Hybrid Visualization Routing
Three layers, ordered by cost:

**Layer 1 — Deterministic pre-filters (no LLM):**
- 0 rows → `table`
- 1 row × 1 col → `metric` (rendered as `st.metric`)
- 1 row × N cols → `table`
- > 15 rows → `table`

**Layer 2 — Keyword intent classifier (no LLM):**
```python
INTENT_HINTS = {
    "tendência":  ["tendência", "evolução", "histórico", "por mês", "por ano"],
    "lista":      ["liste", "quais são", "mostre", "exiba"],
    "ranking":    ["top", "maior", "menor", "mais", "menos", "ranking"],
    "proporção":  ["proporção", "percentual", "%", "distribuição"],
    "contagem":   ["quantos", "quantidade", "número de", "total de"],
    "comparação": ["comparar", "versus", "vs", "diferença entre"],
}
```

**Layer 3 — Cheap LLM** (only for ambiguous 2-15 row results):
Receives `intent_hint + n_rows + columns + numeric_cols + date_cols + sample`. Returns structured `viz_spec` JSON.

Supported viz types: `bar`, `line`, `pie`, `table`, `metric`.

### 4. Per-Node Latency Trace with % Breakdown
Every node tracks its own elapsed time with `time.perf_counter()`. The UI enriches the trace with percentage of total pipeline time:

```
✔ Schema descoberto (0.01s — 0%)
✔ Plano criado (2.34s — 37%)
✔ SQL gerado (1.87s — 30%)
✔ Query executada — 5 linhas (0.00s — 0%)
✔ Visualização: bar — intenção: ranking (0.94s — 15%)
✔ Resposta formatada (1.12s — 18%)
```

### 5. Per-Banker CX Logging (JSONL)
Every query run is appended to `data/logs/<banker_id>.jsonl`. Each record:

```json
{
  "timestamp": "2026-06-26T14:32:01.123456",
  "banker_id": "banker_01",
  "question": "Liste os 5 estados com maior número de clientes...",
  "intent": "ranking",
  "sql": "SELECT estado, COUNT(*) ...",
  "success": true,
  "retries": 0,
  "total_latency_s": 6.28,
  "viz_type": "bar",
  "row_count": 5
}
```

The `extra: dict` field accepts arbitrary new keys (token counts, model name, cost) without schema migration.

**Aggregation functions:**
- `get_banker_metrics(banker_id)` → success rate, avg latency, top intents, recent history
- `get_all_bankers_df()` → merges all bankers into one DataFrame, ready for ML

**Upgrade path:**
```python
# When ready:
get_all_bankers_df().to_parquet("logs/all.parquet")

# Query with DuckDB (no server, no cost):
duckdb.query("SELECT banker_id, AVG(total_latency_s) FROM 'logs/all.parquet' GROUP BY 1")
```

### 6. Provider-Agnostic LLM Factory
Switch models via a single env var. No code changes:

```bash
LLM_PROVIDER=gemini   # Google AI Studio free tier
LLM_PROVIDER=azure    # Azure OpenAI GPT-4o (if credits available)
```

### 7. PT-BR Language Lock
All prompts, trace strings, UI labels, and LLM outputs are in Brazilian Portuguese. Enforced at the prompt level on every node.

### 8. Streamlit UI
- Sidebar: banker ID input, live CX metrics panel, schema explorer
- Main: example questions, chat interface, reasoning trace expander, Plotly charts
- Chat history persists in `st.session_state` across questions in the same session
